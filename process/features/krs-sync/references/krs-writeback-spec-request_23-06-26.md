# เอกสารขอข้อมูลการเชื่อมต่อ (Integration Spec Request) — เขียนสต็อกกลับเข้า KRS

**ถึง:** ทีมผู้พัฒนา/ผู้ดูแลระบบ KRS (db_ACC_SNP)
**จาก:** ทีมพัฒนา KRS POS (ระบบขายหน้าร้าน)
**วันที่:** 23/06/2026
**เรื่อง:** ขอสเปกช่องทางที่ "รองรับอย่างเป็นทางการ" สำหรับให้ POS ลงรายการตัดสต็อก (และกลับรายการ) เข้าสู่ KRS

> เอกสารนี้เป็น **คำขอข้อมูล/สเปก** เพื่อให้เราพัฒนาได้อย่างถูกต้องและปลอดภัย
> **เราจะไม่ INSERT/UPDATE ตรงเข้าตารางของ ERP เอง** เพราะระบบบัญชี/ต้นทุน/เลขเอกสารถูกคำนวณโดย KRS — ขอช่องทางที่ทาง KRS รองรับเท่านั้น

---

## 1. บริบทปัจจุบัน (สิ่งที่ทำได้แล้ว — อ่านอย่างเดียว)

POS เชื่อมต่อ KRS (`db_ACC_SNP`, MS SQL Server) ผ่าน `mssql` แบบ **อ่านอย่างเดียว** และใช้งานได้จริงแล้ว:

| สิ่งที่อ่าน | แหล่ง KRS | ใช้ทำอะไร |
|---|---|---|
| รายการสินค้า | `dbo.InventoryItem` (ItemCode, ItemName, Saleprice1, BarCode, ItemTypename, IsActive) | สร้าง/อัปเดตสินค้าใน POS (`Product`) — sku ของ POS = `ItemCode` |
| ยอดคงเหลือ | `dbo.sp_Onhand(@ItemCode, @Date, @Warehouse)` → `(ItemCode, Balqty)` | เทียบสต็อก POS ↔ KRS และตั้งยอดตั้งต้น |

เราเข้าใจว่า `sp_Onhand` คำนวณจาก `InventoryFlowHdr ⋈ InventoryFlowDtl` (`SUM(InOut × MainQuantity)`) เฉพาะเอกสาร `Approved = 1 AND IsClosed <> 1` — จึงทราบว่าการเคลื่อนไหวสต็อกถูกบันทึกเป็น **เอกสาร InventoryFlow** ที่มีกระบวนการอนุมัติ ไม่ใช่การแก้ยอดในตารางตรงๆ

---

## 2. สิ่งที่ต้องการทำ (ขอบเขตที่ชัดเจน)

โมเดลการแบ่งความเป็นเจ้าของข้อมูล (ตกลงภายในทีม POS แล้ว) = **"แยกตามชนิดการเคลื่อนไหว"**:

- **รับเข้า / ปรับยอด / ซื้อ** → **KRS เป็นเจ้าของ** POS แค่ "ดึง" มาแสดง (ทำได้แล้วผ่าน `sp_Onhand`) — *ไม่ต้องการเขียนกลับส่วนนี้*
- **การขายหน้าร้าน (ตัดสต็อกออก)** → **POS เป็นเจ้าของ** และต้อง **ส่งรายการตัดสต็อกกลับเข้า KRS**

ดังนั้นคำขอนี้จำกัดเฉพาะ **2 เหตุการณ์**:
1. **ตัดสต็อกจากการขาย** (stock-out) — เมื่อ POS ขายสินค้า
2. **กลับรายการ** (reversal) — เมื่อมีการคืนสินค้า/ยกเลิกบิล ต้องคืนสต็อกกลับ

> เราจะส่งเป็น "การเคลื่อนไหว (movement)" ไม่ใช่ "ยอดคงเหลือสุดท้าย" — เพื่อไม่ให้ทับยอดที่ KRS คำนวณเอง

## 3. คำถาม/ข้อมูลที่ขอ (กรุณาตอบเป็นข้อ)

### 3.1 ช่องทางการเขียน (สำคัญที่สุด)
- [ ] ให้ POS ลงรายการตัดสต็อกเข้า KRS ผ่านช่องทางไหนที่ **รองรับอย่างเป็นทางการ**?
  - (ก) **Stored Procedure** (เช่น `sp_PostStockMovement` / `sp_CreateInventoryFlow` …) — ขอชื่อ + พารามิเตอร์ทั้งหมด
  - (ข) **ตาราง staging/import** ที่ KRS มี job ดูดเข้าให้ — ขอชื่อตาราง + โครงสร้าง + รอบการประมวลผล
  - (ค) **API/Web Service** — ขอ endpoint + รูปแบบ payload + การยืนยันตัวตน
  - (ง) อื่นๆ (โปรดระบุ)

### 3.2 ตัวอย่างที่ถูกต้อง 1 ชุด
- [ ] ขอ **ตัวอย่างจริงของการ "ตัดสต็อกจากการขาย" 1 รายการ** ที่ valid — ระบุทุกฟิลด์ที่ต้องส่ง พร้อมค่าตัวอย่าง
  (เช่น ตัดสินค้า `ItemCode = F01-0001` จำนวน `2` ออกจากคลัง `WHFG` วันที่ 23/06/2026)

### 3.3 เลขเอกสาร (TransactionNo / VoucherNo)
- [ ] ใครเป็นผู้ออกเลข — **POS ออกเอง** หรือ **KRS ออกให้**?
- [ ] ถ้า POS ต้องออกเอง: รูปแบบ/กติกาเลขที่รัน (running number) เป็นอย่างไร? มีตาราง/ขั้นตอนจองเลขไหม (เช่น `RunningNumber`)?
- [ ] เลขชุดไหนสงวนไว้ให้ POS ใช้ เพื่อไม่ชนกับเอกสารที่สร้างใน KRS เอง?

### 3.4 การอนุมัติ/ผลต่อยอด (posting)
- [ ] รายการที่ POS ส่งเข้าไป **มีผลต่อยอดทันที** หรือต้องรอ **อนุมัติ (Approved = 1)** ก่อน?
- [ ] ถ้าต้องอนุมัติ — POS ตั้ง `Approved = 1` ได้เลย หรือเป็นขั้นตอนแยกฝั่ง KRS?
- [ ] `sp_Onhand` นับเฉพาะ `Approved = 1 AND IsClosed <> 1` — เพื่อให้ยอดตรงกัน รายการจาก POS ต้องอยู่สถานะใด?

### 3.5 การกันรายการซ้ำ (Idempotency) — จำเป็นมาก
- [ ] มีฟิลด์ใดที่ใช้ **กันการลงซ้ำ** ได้ (unique reference)? เพราะ POS จะ retry อัตโนมัติเมื่อเครือข่ายสะดุด — การส่งซ้ำต้อง **ไม่** ทำให้สต็อกถูกตัดซ้ำ
- [ ] เรามี `orderNumber` ของ POS เป็น reference — มีฟิลด์ฝั่ง KRS ให้เก็บ external reference นี้ไหม (เช่น Remark/RefNo/SourceDoc)?

### 3.6 การจับคู่ข้อมูลหลัก (mapping)
- [ ] **สินค้า:** POS `sku` = KRS `ItemCode` ใช่ไหม (ตรงตัว)?
- [ ] **คลัง:** ค่าที่ต้องส่งในฟิลด์ Warehouse — รหัสคลังหน้าร้านคืออะไร (เช่น `WHFG`)? มีหลายคลังไหม?
- [ ] **เครื่องหมาย InOut:** ตัดออก (ขาย) ใช้ค่าใด (เช่น `-1`) และรับคืน (reversal) ใช้ค่าใด (`+1`)? `MainQuantity` ใส่เป็นบวกเสมอหรือไม่?

### 3.7 ต้นทุน (costing)
- [ ] การตัดสต็อกจากการขายต้อง **ส่งต้นทุนมาด้วยไหม** หรือ **KRS คำนวณเอง** (เช่น standard/FIFO/average)?
- [ ] ถ้าต้องส่ง — ใช้ฐานต้นทุนใด และฟิลด์ไหน?

### 3.8 การกลับรายการ (คืนสินค้า / ยกเลิกบิล)
- [ ] วิธีที่ถูกต้องในการ **คืนสต็อก** กลับเข้า KRS — เอกสารกลับรายการแบบเดียวกับ §3.1 แต่ InOut ตรงข้าม หรือมีกลไกเฉพาะ?
- [ ] ต้องอ้างอิงเอกสารต้นทาง (เลขบิลขายเดิม) ไหม?

### 3.9 ฟิลด์วันที่/เวลา
- [ ] ฟิลด์ใดคือวันที่มีผลต่อยอด (เราเห็น `InOutDate`)? รูปแบบและ time zone?

### 3.10 สภาพแวดล้อมทดสอบ (Sandbox) — บังคับก่อน go-live
- [ ] ขอ **ฐานทดสอบแยกจากของจริง** + บัญชีผู้ใช้ทดสอบ เพื่อทดสอบการเขียนโดย **ไม่กระทบบัญชีจริง**
- [ ] เราจะไม่ทดสอบการเขียนใดๆ บนฐาน production จนกว่าจะ verify บน sandbox ครบ

### 3.11 สิทธิ์/บัญชีเชื่อมต่อ (ความปลอดภัย)
- [ ] ปัจจุบันเชื่อมด้วยบัญชี `sa` — ขอให้ออก **บัญชีเฉพาะกิจสิทธิ์จำกัด** สำหรับ POS (อ่าน `InventoryItem`/`sp_Onhand` + สิทธิ์เรียกช่องทางเขียนตาม §3.1 เท่านั้น) เพื่อ defense-in-depth

### 3.12 ปริมาณ/ความถี่ & ข้อผิดพลาด
- [ ] รองรับการส่งทีละรายการ (ต่อบิล) แบบ near-realtime ได้ไหม หรือควรส่งเป็นชุด (batch)? มีเพดานความถี่ไหม?
- [ ] รูปแบบ error ที่ส่งกลับเมื่อสำเร็จ/ล้มเหลว เป็นอย่างไร (เพื่อให้ POS รู้ว่าควร retry หรือหยุด)?

## 4. สิ่งที่ฝั่ง POS รับประกัน (เพื่อความสบายใจของทีม KRS)
- เขียนผ่าน **ช่องทางที่ KRS กำหนดเท่านั้น** — ไม่แตะตารางตรง
- **ส่งทุกบิลแบบกันซ้ำ** ด้วย reference + retry ปลอดภัย (idempotent)
- **fail-open:** ถ้า KRS ไม่ตอบ การขายหน้าร้านดำเนินต่อได้ รายการจะถูกพักไว้แล้วส่งซ้ำภายหลัง (ไม่มีการ "ตัดซ้ำ")
- ทดสอบบน **sandbox** จนครบก่อนเปิดใช้จริง

## 5. รูปแบบคำตอบที่อยากได้ (สรุป)
1. ช่องทางเขียน (proc/table/API) + สเปกฟิลด์ครบ
2. ตัวอย่าง valid 1 ชุด (ขาย-ตัดออก) + 1 ชุด (กลับรายการ)
3. กติกาเลขเอกสาร + การอนุมัติ + ฟิลด์กันซ้ำ
4. ข้อมูล sandbox + บัญชีสิทธิ์จำกัด

---

### ภาคผนวก — สิ่งที่เราสังเกตจากสคีมา (เพื่อช่วยทีม KRS อ้างอิง ไม่ใช่ข้อสรุป)
- `InventoryFlowHdr` มีฟิลด์เช่น `TranSactionno`, `VoucherNo`, `InOutDate`, `Approved`, `IsClosed`, `Warehouse(ใน Dtl)`
- `InventoryFlowDtl` มี `Transactionno`, `VoucherNo`, `ItemCode`, `Warehouse`, `InOut`, `MainQuantity`
- `sp_Onhand` = `SUM(InOut × MainQuantity)` GROUP BY `ItemCode` เฉพาะ `Approved=1 AND IsClosed<>1`

> หากการลงเอกสาร InventoryFlow โดยตรงไม่ใช่วิธีที่แนะนำ โปรดชี้ช่องทางที่ถูกต้องแทน — เรายินดีปรับตามที่ KRS รองรับ
